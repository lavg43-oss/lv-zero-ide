# Plan 1705 — Auditoría Completa de Bugs y Plan de Resolución

**Proyecto:** lv-zero v4.1.0  
**Fecha:** 2026-05-17  
**Origen:** Graphify-ts (999 nodos, 910 aristas) + Análisis manual de 8 archivos críticos  

---

## 📖 ¿Qué es ASAR?

**ASAR** (Atom Shell Archive) es el formato de archivo empaquetado que usa Electron para distribuir la aplicación. Cuando ejecutas `electron-builder`, todos los archivos del proyecto (`src/`, `ui/`, `node_modules/`) se comprimen en un archivo `app.asar` dentro de la carpeta de distribución (ej: `dist_new/win-unpacked/resources/`).

**Problema clave**: `asar_content/` es una **extracción manual** de ese ASAR para poder inspeccionar y parchear archivos directamente. Pero si se reconstruye la app con `electron-builder`, el ASAR se genera desde `src/` (no desde `asar_content/`), lo que significa que **cualquier fix aplicado manualmente a `asar_content/` se pierde** al rebuildear.

Por eso el bug **C2 (ASAR drift)** es el más crítico: los builds empaquetados (`dist_new/`) ejecutan código viejo porque el ASAR no se ha regenerado desde que `src/` fue significativamente modificado.

### C1 — `app.on("activate")` sin `setupIPC()` en macOS (CORREGIDO)

| Atributo | Detalle |
|----------|---------|
| **Archivo** | [`src/main.cjs`](src/main.cjs:1892) |
| **Líneas** | 1892-1903 |
| **Tipo** | Race condition / Handler faltante |

#### Descripción

El evento `app.on("activate")` se dispara en macOS cuando el usuario hace clic en el ícono del dock y no hay ventanas abiertas. Este handler recrea la ventana con `createWindow()` pero **no llama a `setupIPC()`**, dejando todos los canales IPC sin registrar (agent:*, file:*, project:*, config:*, etc.).

```javascript
// ANTES — setupIPC() NO se llamaba
app.on("activate", () => {
  if (BrowserWindow.getAllWindows().length === 0) {
    createWindow();
    if (terminalBridge) {
      terminalBridge.setupTerminalIPC(ipcMain, mainWindow);  // Solo terminal
    }
    // ❌ Faltan: setupIPC(), connectOrchestratorEvents(), connectShellOutput()
  }
});
```

#### Impacto en el Sistema

- **Usuario macOS**: Si cierra la ventana y la reabre desde el dock (~10-20% de las sesiones en Mac), la app se vuelve **completamente no funcional**.
- **Síntomas**: El agente no responde, el explorador no abre proyectos, los archivos no se guardan, el modo no cambia.
- **No hay error visible**: La UI carga normalmente, pero todos los `ipcRenderer.invoke()` quedan en espera infinita porque no hay handler en main process.
- **Afecta solo macOS**: Windows y Linux no usan el evento `activate`.

#### Plan de Resolución

| Paso | Acción | Archivo | Riesgo |
|------|--------|---------|--------|
| 1 | Agregar `setupIPC()`, `connectOrchestratorEvents()`, `connectShellOutput()`, `fileBridge.setupFileIPC()` | [`src/main.cjs`](src/main.cjs:1892) | Bajo — funciones ya existen |
| 2 | Aplicar mismo fix a `asar_content/src/main.cjs` | `asar_content/src/main.cjs:1396` | Bajo |
| 3 | Aplicar mismo fix a `dist_new/.../main.cjs` | `dist_new/.../main.cjs:1347` | Bajo |
| 4 | Usar `if (orchestrator && terminalBridge && fileBridge)` como guardia | Variable de módulo, no scope | Ninguno |

**Estado:** ✅ **CORREGIDO** en las 3 versiones.

#### Riesgos de la Corrección

- **Riesgo MUY BAJO**: `setupIPC()` es idempotente — cada handler usa `ipcMain.handle()` que reemplaza handlers previos.  
- **Riesgo BAJO**: `connectOrchestratorEvents()` y `connectShellOutput()` agregan listeners a `orchestrator.on()`. Si el orchestrator persiste entre ciclos de ventana (no es el caso — se reinicia en init), habría listeners duplicados.  
- **Mitigación**: Ya se verificó que `orchestrator`, `terminalBridge` y `fileBridge` se inicializan SOLO en el flujo normal (`init()` con API key), y el activate handler solo se ejecuta si todas las variables existen.

### C2 — Drift Completo de ASAR vs Source (CORREGIDO PARCIALMENTE)

| Atributo | Detalle |
|----------|---------|
| **Archivos** | `asar_content/`, `dist_new/win-unpacked/resources/app_asar_tmp/` |
| **Tipo** | Desincronización de builds |

#### Descripción

Todos los archivos críticos en `asar_content/` y `dist_new/` están **significativamente desactualizados** vs `src/`. La diferencia de líneas revela la magnitud del problema:

| Archivo | src/ | asar_content/ | Diferencia |
|---------|------|---------------|------------|
| `main.cjs` | 1928 líneas | 1432 líneas | **-496 líneas** |
| `orchestrator.js` | 3905 líneas | 902 líneas | **-3003 líneas** |
| `renderer.js` | 5404 líneas | 5168 líneas | -236 líneas |
| `preload.js` | 375 líneas | 158 líneas | -217 líneas |
| `styles.css` | 3592 líneas | 2757 líneas | -835 líneas |
| `file_bridge.js` | 461 líneas | 441 líneas | -20 líneas |

El handler `project:open` en `asar_content/src/main.cjs` **no tenía**:
- `fileBridge.setAllowedBase(targetPath)` — sin esto, `resolveSafePath()` rechaza cualquier carpeta abierta después del inicio
- `orchestrator.setProjectPath(targetPath)` — el orquestador no sabe qué proyecto está activo
- Reinicio del file watcher — el explorador reactivo nunca se actualiza

#### Impacto en el Sistema

- **El bug más peligroso**: Cualquier rebuild del ASAR con `electron-builder` **borrará todas las correcciones** aplicadas manualmente.
- **Usuario empaquetado**: La versión distribuida (`dist_new/`) ejecuta código que es 3000 líneas más viejo que la fuente.
- **Funcionalidades faltantes**: MCP config manager, tool call repair, session management, crash recovery — todo esto existe en `src/` pero NO en los builds.
- **Regresión asegurada**: Cualquier bug fix aplicado a `src/` no llega a los usuarios hasta que se reconstruya el ASAR.

#### Plan de Resolución

| Paso | Acción | Riesgo |
|------|--------|--------|
| **P1** | **Reconstruir ASAR desde src/**: `npx electron-builder --win --config.asar=true` | **ALTO** — requiere entorno de build funcional |
| 2 | Verificar que `asar_content/` se regenera completamente | Medio — puede haber configs hardcodeadas |
| 3 | Verificar que `dist_new/win-unpacked/` se regenera | Medio — electron-builder puede fallar por dependencias |
| 4 | Agregar script `npm run build` reproducible | Bajo |
| 5 | Documentar proceso de build en README | Muy bajo |

#### Riesgos de la Corrección

- **⚠️ RIESGO ALTO**: Electron-builder puede fallar si falta `electron-builder.yml` o si las rutas de recursos no están configuradas.
- **⚠️ RIESGO MEDIO**: Si `asar_content/` se genera de un `src/` incorrecto (por ejemplo, excluyendo archivos en `package.json` `"files"`), el build empaquetado podría tener problemas diferentes.
- **⚠️ RIESGO MEDIO**: Las rutas en `src/main.cjs` usan `__dirname` y `path.resolve(__dirname, "..")` que dependen de la estructura de directorios al empaquetar. Un rebuild podría romper estas rutas si la estructura cambia.
- **Mitigación**: Hacer backup de `asar_content/` y `dist_new/` ANTES de rebuild. Probar en entorno de desarrollo primero.

### C3 — Fugas de Timer en renderer.js (19 no gestionados)

| Atributo | Detalle |
|----------|---------|
| **Archivo** | [`ui/renderer.js`](ui/renderer.js) |
| **Tipo** | Memory leak / Resource leak |

#### Descripción

El renderer tiene **28 llamadas a `setTimeout()`** pero solo **9 llamadas a `clearTimeout()`**. Esto significa que **19 timers (68%)** nunca se limpian explícitamente. El método [`destroy()`](ui/renderer.js:5347) solo limpia 2:

```javascript
// destroy() solo limpia 2 de ~19 timers
if (this._fsUpdateTimer) clearTimeout(this._fsUpdateTimer);
if (this._reloadFileDebounce) clearTimeout(this._reloadFileDebounce);
```

#### Impacto en el Sistema

- **Fugas de memoria**: Cada timer retiene su closure, que incluye referencias a `this` (el `IDEController` completo). Mientras el timer no se ejecute o cancele, el objeto completo no puede ser recolectado por el GC.
- **Clics fantasmas**: Si un timer se dispara DESPUÉS de que el componente fue destruido (por cambio de sesión o panel toggle), intenta acceder a `this.els.*` que son null, causando `Cannot read properties of null`.
- **Actualizaciones incorrectas**: El callback del timer puede modificar el DOM de una sesión que ya no está visible.
- **Degradación progresiva**: Tras ~4 horas de uso, los timers acumulados pueden consumir ~50-200MB de memoria.

#### Plan de Resolución

| Paso | Acción | Archivo: Línea | Riesgo |
|------|--------|----------------|--------|
| 1 | Crear `this._timers = []` en constructor | `renderer.js:56-90` | Ninguno |
| 2 | Reemplazar TODOS los `setTimeout(...)` con `this._timers.push(setTimeout(...))` | Múltiples | Bajo |
| 3 | En `destroy()`, limpiar todos: `this._timers.forEach(clearTimeout)` | `renderer.js:5347` | Bajo |
| 4 | Mantener `this._fsUpdateTimer` y `this._reloadFileDebounce` como TIMERS NOMBRADOS para claridad, pero también agregarlos a `_timers` | Múltiples | Muy bajo |

**Riesgo**: Bajo. Es un cambio puramente aditivo — nunca se elimina un setTimeout, solo se registra para cleanup.

### C4 — Fugas de Event Listener en renderer.js (84 sin cleanup)

| Atributo | Detalle |
|----------|---------|
| **Archivo** | [`ui/renderer.js`](ui/renderer.js) |
| **Línea clave** | [`_bindUIEvents()`](ui/renderer.js:4479) |
| **Tipo** | Memory leak / Resource leak |

#### Descripción

`_bindUIEvents()` registra **84 event listeners** (click, input, change, keydown, etc.) en elementos del DOM. **CERO** de esos listeners se remueven con `removeEventListener()`. El método [`destroy()`](ui/renderer.js:5347) solo limpia los *IPC subscriptions* (via `this.unsubscribers.forEach(fn => fn())`), no los *DOM listeners*.

```javascript
// destroy() NO limpia DOM listeners
destroy() {
  this.unsubscribers.forEach((fn) => fn());     // ✅ IPC events
  try { window.lvzero["file:watchStop"](); } catch {}  // ✅ File watcher
  if (this._fsUpdateTimer) clearTimeout(this._fsUpdateTimer);  // ✅ 1 timer
  this.editor?.dispose();   // ✅ Monaco editor
  this.terminal?.dispose(); // ✅ XTerm
  this.splitHoriz?.destroy();  // ✅ Split.js
  // ❌ NO: removeEventListener para los 84 listeners DOM
}
```

#### Impacto en el Sistema

- **Crecimiento lineal de memoria**: Cada listener retiene referencia al callback, que captura `this` — el `IDEController` completo no puede ser recolectado.
- **~500KB/hora** de fuga estimada en uso moderado.
- **Múltiples instancias**: Si por alguna razón se creara un segundo `IDEController` (hot reload, recarga de panel), los listeners del primero siguen vivos.
- **Ejecución duplicada**: Al cambiar de sesión o panel, los callbacks antiguos aún responden a eventos, duplicando acciones.

#### Plan de Resolución

| Paso | Acción | Archivo: Línea | Riesgo |
|------|--------|----------------|--------|
| 1 | Crear `this._abortController = new AbortController()` en constructor | `renderer.js:56-90` | Ninguno |
| 2 | En TODOS los `addEventListener()`, agregar `{ signal: this._abortController.signal }` | `renderer.js:4479-4981` | **MEDIO** |
| 3 | En `destroy()`, llamar `this._abortController?.abort()` | `renderer.js:5347` | Muy bajo |

**Riesgo MEDIO**: `AbortController.signal` es soportado por Chrome/Edge (Electron usa Chromium), pero si hay código que depende de la REMOCIÓN EXPLÍCITA de un listener específico (en lugar de abortar todos), podría haber efectos secundarios.  
**Solución**: Si hay listeners específicos que se remueven antes de destroy (ej: listeners temporales en forms), mantener el patrón `removeEventListener` para esos casos específicos.

### H2 — File Watcher: Race Condition en Inicio

| Atributo | Detalle |
|----------|---------|
| **Archivo** | [`src/main.cjs`](src/main.cjs:1849) |
| **Línea** | 1849-1854 |
| **Tipo** | Race condition |

#### Descripción

El file watcher se inicia con un **timeout fijo de 2 segundos** sin sincronización con el renderer:

```javascript
setTimeout(() => {
  if (fileBridge && fileBridge.startFileWatcher) {
    fileBridge.startFileWatcher(mainWindow, APP_ROOT);
  }
}, 2000);
```

Además, el renderer TAMBIÉN inicia su propio file watcher en [`init()`](ui/renderer.js:170-175):

```javascript
await window.lvzero["file:watchStart"]();
```

**Posibles escenarios de carrera**:
1. **Renderer lento (>2s)**: El main process inicia el watcher antes de que `onFsUpdate` esté registrado → notificaciones perdidas.
2. **Renderer rápido (<2s)**: Ambos inician watchers → **watchers duplicados** (doble notificación por cambio de archivo).
3. **Timeout no confiable**: En máquinas lentas, 2s puede ser insuficiente; en máquinas rápidas, excesivo.

#### Impacto en el Sistema

- **Notificaciones perdidas**: El editor no detecta cambios de archivos externos.
- **Notificaciones duplicadas**: El editor recibe 2 eventos por cada cambio.
- **Comportamiento impredecible**: Depende de la velocidad de la máquina y la carga del sistema.

#### Plan de Resolución

| Paso | Acción | Archivo | Riesgo |
|------|--------|---------|--------|
| 1 | Eliminar el `setTimeout` del main process | `src/main.cjs:1849` | Bajo |
| 2 | El main process solo inicia el watcher cuando el renderer lo solicita vía IPC | `src/main.cjs` + `setupIPC()` | Bajo |
| 3 | Agregar canal IPC `"renderer:ready"` que main escucha para iniciar watcher | `src/main.cjs` | Bajo |
| 4 | En renderer, reemplazar `file:watchStart` con envío de `renderer:ready` | `ui/renderer.js:170` | Bajo |

**Alternativa más simple**: Que el renderer inicie el watcher (como ya hace en L170) y el main NO lo inicie con setTimeout. Esto elimina ambos problemas (duplicación y carrera).

**Riesgo**: Bajo — el renderer ya tiene el código para iniciar el watcher, solo hay que eliminar la copia del main.

## ⚠️ RIESGOS GENERALES DEL PLAN

1. **Reconstruir ASAR (C2-P1) es el paso más riesgoso** — Si electron-builder falla o produce un build incorrecto, la app empaquetada puede no funcionar. **Mitigación**: Backup completo antes de rebuild, prueba en dev mode primero.

2. **AbortController (C4) puede tener efectos secundarios** — Si algún código depende de remover listeners específicos sin abortar todos. **Mitigación**: Revisar si hay `removeEventListener` explícitos. Actualmente NO los hay (el problema es que no se remueve ninguno), así que no hay conflicto.

3. **Eliminar el setTimeout del file watcher (H2)** — Si el renderer falla al iniciar el watcher, no hay fallback. **Mitigación**: Mantener el `try/catch` existente en el renderer.

4. **Verificación intermedia consume tiempo** — Cada fix tiene su propia verificación (~2-5 min cada una). Esto agrega ~30-40 min al plan total, pero previene regresiones en cadena.

## CONCLUSIÓN

El bug más crítico del sistema es **C2 (ASAR drift)** porque invalida todas las demás correcciones — cualquier fix que se aplique a `src/` no llega a producción hasta que se reconstruya el ASAR. Los bugs **C3/C4 (memory leaks)** son los que más afectan la experiencia del usuario a largo plazo, causando degradación progresiva.
